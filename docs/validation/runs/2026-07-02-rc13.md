### BOOT-02 — Fresh A2DP pairing — **FAIL → fixed → partially revalidated**

- **Expected:** `POST /audio/pair` on a never-bonded speaker discovers, pairs,
  persists `audio_sink_address`, `audio_ready` goes true.
- **Observed (as shipped in RC13):** pairing crashes ~0.3 s after the FDDF
  address is derived: `InterfaceNotFoundError: org.bluez.Device1`, surfaced to
  the Portal as `Unexpected error: interface not found…`. **Root cause:**
  `BluezClient.pair()` calls `Pair()` on a D-Bus object BlueZ never created —
  the BR/EDR address derived from the FDDF LE advertisement has never been
  *seen* by BlueZ on the BR/EDR transport, so no `Device1` object exists.
  Dev hardware masked this (a device object left over from earlier manual
  pairing). This is exactly the never-hardware-verified path M19 flagged;
  it can never have worked on a clean image.
- **Hardware facts established:** JBL answers BR/EDR inquiry **only in
  pairing mode** — not in steady-on, not in a 40 s window after power-on
  (verified with power-cycle + `scan bredr`: 14 other devices found, JBL
  absent). `bluetoothd` runs without `-E`, so `Adapter1.ConnectDevice` is
  unavailable.
- **Fix (deployed + in repo):** `BluezClient.wait_for_device()` — after FDDF
  address derivation, run a BR/EDR-transport discovery and wait for the
  `Device1` object to appear (speaker must be in pairing mode, which the UX
  already instructs); `pair()` now raises `PairingFailedError` instead of
  leaking `InterfaceNotFoundError`; `is_connected()` hardened for vanished
  device objects. +3 unit tests.
- **Revalidated (autonomous half):** pair with speaker on but *not* in
  pairing mode → FDDF address resolved instantly, 30 s BR/EDR window, clean
  `WARNING … not in pairing mode?`, Portal-facing state `failed` with
  actionable message, no traceback. ✔
- **Outstanding:** happy path requires the physical pairing-mode button press
  → **human batch**.
- **Follow-up:** consider surfacing "put the speaker in pairing mode first"
  in the Portal *before* the attempt, since the window is short.

### HOST-03 — bluetooth restart under companion — **FAIL → fixed → PASS**

- **Observed (as shipped):** after `systemctl restart bluetooth` the daemon
  **never notices**: `/health`/`/speaker` report `connected: true` forever
  (zombie), every power command 503s, **no self-recovery** — appliance dead
  until companion restart. Two stacked causes:
  1. bleak's disconnect callback never fires — a restarting bluetoothd exits
     without emitting `InterfacesRemoved`, so `drain_until_disconnect()`
     waits forever;
  2. bleak's `is_connected` is *cached* state — it also stays `true`, so
     passive checks can't detect the zombie either (verified empirically:
     a first-iteration fix that cross-checked `is_connected` detected
     nothing). Only a real ATT operation fails.
- **Fix (deployed + in repo):** `PartyBoxDevice.verify_connection()` — a
  benign ATT write round-trip (firmware-version request, reply drained);
  `DeviceManager._drain_with_health_check()` probes every 15 s while
  draining and treats probe failure as connection loss. +2 SDK unit tests.
- **Revalidated:** bluetooth restart → probe failure detected in ~9 s →
  `connection lost, will reconnect` → reconnected +20 s → **full GATT
  round-trip (power 204) at +22 s**, fully autonomous. ✔
- **Also protects against:** any silent BLE stack wedge where the disconnect
  callback is lost — the 15 s probe bounds zombie lifetime in all cases.

### HOST-04 — WirePlumber restart — **PASS** (pre-pairing scope)

`systemctl --user restart wireplumber` → both wireplumber+pipewire active,
companion unaffected. Audio-graph rebuild dimension re-tested post-pairing.

## INC-1 — Live incident during dogfooding (2026-07-03 ~14:16) + STREAM-02

**Report (operator):** speaker unplugged all day → plugged in, powered on
via Portal, "Spotify Connect auto-connected", music played, Portal's
Bluetooth Audio card stuck yellow "Connecting…", music stopped ~5 min in
and could not be restarted.

**Reconstructed timeline (journal + soak log):** speaker unreachable until
14:10:38 (BLE connect, attempt 1695 — appliance had scanned calmly all
day, RSS flat, no leak). 14:11:16 A2DP connected + librespot started.
Operator's playback ran ~14:12–14:16, then the librespot session went
inactive with **zero trace at INFO** — process alive, A2DP up,
`audio_ready` true throughout.

**Root-cause analysis (high confidence, not conclusive):** after enabling
DEBUG and re-running the same flow, librespot 0.8 logged
`ERROR librespot_connect::context_resolver] setup of state failed: Invalid
state {the current track couldn't be resolved from the transfer state}`
plus `WARN … context is not available` around session transfer and a
`spotify:station:album:…` (autoplay radio) context. The incident pattern —
session auto-transferred to a freshly registered device, a few tracks
play, playback dies when the context/autoplay resolution fails, restart of
the session fixes it — matches this librespot 0.8 connect-state failure
class. Retry under instrumentation: 60+ min playback incl. pause/resume,
zero errors, not reproduced.

**Defects found and fixed (commit `fb42637`):**
1. librespot ERROR/WARN stderr was logged at DEBUG — the incident's cause
   was invisible at default log level. Now re-logged at matching levels.
2. `active` playback detection never fired on librespot 0.8 (log format
   changed; `/api/v1/spotify` reported `active:false` during playback).
   Fixed with hardware-captured pattern. **Also captured: pause/stop emit
   NO stderr at 0.8's default verbosity** — deactivation is undetectable
   from logs; PipeWire stream state is the robust source (deferred,
   status-display only).
3. Empty A2DP failure message (`err:`) when DBusError has no text — now
   falls back to the D-Bus error name.

**Open (Portal):** whether the Bluetooth Audio card updates from
yellow→green without a page reload needs a targeted check (OBS-12).

**Side effect:** this session doubles as **STREAM-02/03 evidence** — 60+
min of real Spotify Connect playback with track changes and pause/resume,
zero transport errors. Appliance left at `COMPANION_LOG_LEVEL=DEBUG`
(journald volatile, acceptable) until the librespot-level fix is deployed;
revert then.

## INC-2 — Volume level mismatch: quiet music vs. loud native sounds — **confirmed defect**

**Report (operator):** music plays very low from the start; after
compensating with the hardware knob, the speaker's native sounds (power
chime etc.) are far louder than the music.

**Confirmed chain (live inspection):** music is attenuated by two software
stages the native sounds bypass: (1) **PipeWire sink volume 0.40** —
WirePlumber's default for a newly-seen device; nothing in the appliance
ever sets it, so every fresh install ships a permanent 40 % software cut
on music only; (2) librespot softvol (~59 %, wherever the user's Spotify
slider sits, Log(60) taper). The knob (AVRCP absolute volume, observed
live at 96/127) drives the speaker's amplifier — chime at knob level,
music at knob × 0.40 × softvol. Also confirmed: the PipeWire sink volume
does **not** track the AVRCP hardware volume (knob at 0.76 while sink
showed 0.40), and `POST /api/v1/volume` is display-state only (BLE volume
opcode never confirmed — VolumeCapability raises NotImplementedError), so
the M15 "unified volume" is architecturally present but hardware-inert.

**Live remediation:** `wpctl set-volume 76 1.0` — WirePlumber persists
per-device node volume in its state file, so this survives reboots *on
this unit*, but not a reflash.

**Required product fix (M19 punch list, release-blocking for audio UX):**
pin the A2DP sink node to 100 % whenever it appears (AudioService after
connect, or a WirePlumber config drop-in in the image). Longer term (M15
completion): translate librespot volume events → hardware volume via
AVRCP absolute volume (`MediaTransport1.Volume` is writable — observed
live; this sidesteps the missing BLE opcode entirely), pin softvol to
fixed, and make the knob/API/Spotify slider converge on one hardware
volume. The AVRCP path also gives knob-position readback for
`GET /api/v1/volume` (knob events arrive as `MediaTransport1`
`PropertiesChanged` — observed live at DEBUG).
