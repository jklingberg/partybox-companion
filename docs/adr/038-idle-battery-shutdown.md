# ADR-038: Idle Battery Shutdown

**Status:** Accepted

---

## Context

A user reported that Companion left their PartyBox's USB-out power rail — which powers the Pi — running continuously overnight while the speaker sat on battery power, fully draining the speaker's battery. [ADR-033](033-speaker-standby-detection.md) had already addressed a related symptom (the Portal showing "Turn off" on a speaker that was actually already asleep), but that ADR only fixed the *displayed* power state; it did nothing about the Pi's own power draw, and neither ADR-033 nor anything else in this project's ADRs addresses the Pi continuously drawing current from a battery that isn't mains-connected.

The product goal, as stated by the user: while mains-connected, Companion should run indefinitely, no exceptions; while on battery with no activity, it should shut itself down after "roughly 15–30 minutes" — given as a range, not a precise figure, since it was originally intended to be a user-tunable Portal setting (see the third follow-up below for why that changed).

Two hardware facts, established through discussion and cross-referencing existing ADRs before designing this decision, rule out several designs that would otherwise look attractive:

1. **The PartyBox's USB-out never cuts power on its own.** Not during the speaker's own auto-idle standby — [ADR-033](033-speaker-standby-detection.md) already established the BLE control link (and, by the same token, the USB-out rail) stays live through standby. And not even when Companion issues a BLE `power.turn_off()` command: [ADR-034](034-power-command-reconnect-wait.md) hardware-confirmed that command only resets the speaker's own control MCU (BLE drops for ~15s, then `DeviceManager` reconnects and keeps running) — proof the Pi's power was never interrupted by it. The *only* thing that cuts the rail is a manual 10-second hold of the speaker's physical power button. There is no confirmed vendor opcode that replicates this, and per this project's reverse-engineering discipline (`docs/reverse-engineering/guide.md`), nothing is built against an opcode that hasn't been captured on the wire.
2. **Consequently, Companion cannot trigger that hard cutoff itself**, and once it happens (by any means), the whole unit — including the speaker's own BLE radio — is unpowered, so nothing can auto-wake it without a person touching the button again. A literal 0 W shutdown with automatic recovery is not achievable in software alone; it would require new hardware (e.g. a GPIO-controlled relay between the USB-out and the Pi), which is out of scope here.

Given that, the accepted trade-off (explicitly chosen over a software-only "dormant mode" that keeps the OS running and never reaches near-zero power) is: Companion issues a real, OS-level `poweroff` after a sustained idle-on-battery period. This gets power draw down to the Pi board's own quiescent current — a large reduction, but not literally zero — and recovery requires the user to physically power-cycle the speaker (10s hold off, then on), which re-energizes the USB rail and cold-boots the Pi exactly like a fresh install. This mirrors how the appliance already boots today; no new recovery mechanism was designed or built.

### Existing signals were sufficient — no new protocol work

- **Idle**: `StatusSnapshot.speaker_state == "standby"` (`packages/partyboxd/src/partyboxd/device/manager.py`) — the speaker's own MCU has stopped answering control-plane probes ([ADR-033](033-speaker-standby-detection.md)). This is a better idle proxy than tracking Companion's own playback state (Spotify/AirPlay) would be: it's what the hardware itself already does when audio stops, and it requires no new plumbing.
- **Battery vs. mains**: `StatusSnapshot.battery_status.charging_status.on_mains` (`ChargingStatus` in `packages/partybox/src/partybox/protocol/messages.py`), the same field already surfacing as `power_source` on `GET /api/v1/battery`. `charging_status` can legitimately be `None` (a known BLE-fragmentation edge case — see `docs/reverse-engineering/discoveries.md`); this is treated as "power source unknown," never as "on battery."

### The privilege problem, and why D-Bus/polkit instead of sudo

`companion.service` runs with `NoNewPrivileges=true` (`system/systemd/companion.service`). [ADR-028](028-audio-readiness-model.md)'s "Deferred: runtime self-healing needs a privileged broker" section already hit this exact wall for an unrelated recovery feature (a BLE-controller-wedge auto-fix) and explicitly rejected a piecemeal sudoers grant: *"Expanding companion's privileges piecemeal every time a new runtime recovery need appears works against the least-privilege design ... it makes the unprivileged appliance process a growing root-adjacent surface, one grant at a time."* `sudo` is also a structural dead end regardless of that judgment call: `NoNewPrivileges` blocks the setuid escalation `sudo` itself requires, so a sudoers rule for `poweroff` would simply never fire.

(In the course of investigating this, `CLAUDE.md`'s claim of an existing `companion` sudoers grant for restarting WirePlumber was found to be **stale** — no sudoers file exists anywhere in this repository or image. That grant was proposed, never implemented, and the doc was never corrected; already flagged as an M19 punch-list item in `docs/validation/runs/2026-07-02-rc13.md`. Not addressed by this ADR; noted here since it was discovered while researching the same privilege model.)

[ADR-021](021-network-provisioning.md) already established the working alternative for this account: a **polkit rule** granting `companion` D-Bus access, installed by `install.sh`, justified by `companion` being a single-purpose, no-login system account with no other use for elevated access. This decision reuses that mechanism for `systemd-logind`'s `PowerOff` method instead of NetworkManager's API.

## Decision

1. **`PortalConfig.idle_shutdown_minutes: int = 30`** (`packages/companion/src/companion/config_store.py`), `0` disables the feature. Portal-configurable via the existing `GET`/`PUT /api/v1/config` endpoints and a new field in the Settings sheet. Ships **enabled by default** (30 min) — deliberately, matching the product's stated goal of feeling "finished" out of the box rather than requiring the user to discover and opt in. `ge=0` is the only validation constraint; no floor above that, since a low value (2 min) is the useful way to validate the feature on real hardware without waiting half an hour per trial.

2. **`companion.services.login1_dbus.power_off()`** (new module) — a one-shot `dbus-fast` call to `org.freedesktop.login1.Manager.PowerOff(False)` on the system bus. Mirrors `bluez_dbus.py`'s existing "thin transport wrapper, not a service" style (same dependency, same reasoning for avoiding `from __future__ import annotations` — it breaks `dbus-fast`'s signature inference).

3. **`_idle_battery_shutdown()`** (new watcher in `companion/__main__.py`, registered with the `Supervisor`, same shape as the existing `_recheck_audio_on_standby` watcher) — polls `DeviceManager.snapshot()` every 15 s (matching ADR-033's own health-check cadence) rather than reacting purely to `SpeakerStateChangedEvent`: that event only fires on off/standby/on transitions, not on charging-status changes (e.g. the user plugs in mains while the speaker stays in standby), so a periodic snapshot re-check is needed to notice that too. Reads `idle_shutdown_minutes` fresh from `ConfigStore` on every tick, so a Portal-side edit takes effect without a service restart. Requires `speaker_state == "standby"` specifically, not `"off"` — if the BLE link drops entirely, the current power source can no longer be confirmed, so the idle clock resets rather than acting on stale data. Once triggered, does not fire again (the process is about to halt).

4. **New polkit rule**, installed by `install.sh` alongside the existing NetworkManager rule: grants `companion` exactly `org.freedesktop.login1.power-off` and `org.freedesktop.login1.power-off-multiple-sessions` (the latter likely needed since the `pi` user's own systemd session, which `companion.service` depends on for PipeWire, is probably visible to `logind` as a second session). Scoped to these two specific action IDs rather than the whole `org.freedesktop.login1.*` namespace — unlike NetworkManager's action-ID surface (which ADR-021 found to be "fragile ... varies across NM versions," forcing a namespace-wide grant), `logind`'s poweroff-related action IDs are small and stable, so a narrow allowlist is both sufficient and the more defensible scope for a feature that can take the whole appliance offline if triggered wrongly.

## Consequences

**Benefits:**
- Directly fixes the reported incident: a speaker left on battery with no activity no longer bleeds its battery through an idle Pi indefinitely.
- Reuses every existing signal (`speaker_state`, `charging_status`) — no protocol work, no new opcodes.
- The D-Bus/polkit mechanism sidesteps `NoNewPrivileges` entirely rather than fighting it, and the grant is scoped narrower than the precedent (ADR-021) it mirrors.

**Accepted trade-offs:**
- Not literally 0 W — the halted Pi still draws its own quiescent current from the still-live USB rail. Real measurement of that draw was not performed as part of this change; if it turns out to be more significant than assumed, the only further reduction available in software is already exhausted (the OS is fully halted).
- Recovery is manual: the user must physically power-cycle the speaker to bring Companion back online. There is no Portal countdown or warning before the shutdown fires — the whole point is this happens while nobody is around to see it; a live status indicator was considered out of scope for this change and can be added later without any architectural changes if wanted.
- Growing `companion`'s D-Bus privilege footprint by one more grant, which is the exact pattern ADR-028 cautioned against in general. The mitigations here are: the grant is two specific, stable action IDs (not a namespace); the triggering condition is narrow and fully logged before it fires (`log.warning` with the elapsed minutes); and unlike ADR-028's deferred audio-recovery broker, there is no non-privileged fallback available for this feature — accepting the drain was the only alternative.

## Rejected alternatives

- **Software-only "dormant mode"** (stop the BLE connection, stop polling, keep the OS running to auto-recover without any manual step) — this was the first design proposed and initially recommended, since it needs no new privilege grant at all. Rejected by explicit user decision: it cannot get close to 0 W (the Pi board stays fully booted), and the user's stated goal for the product was a real power-off, with a manual recovery step accepted as reasonable.
- **`CAP_SYS_BOOT` + a raw `reboot(2)` syscall**, granted the same way `CAP_NET_BIND_SERVICE` already is (`AmbientCapabilities` in `companion.service`) — rejected: this bypasses the normal systemd shutdown sequence entirely (no clean unmount/sync of the SD card), risking filesystem corruption on every trigger, for no benefit over going through `logind` correctly.
- **A privileged recovery broker** (a separate root-owned process Companion requests actions from), the direction ADR-028 sketched as a possible future path for its own unrelated recovery need — considered, but rejected as unnecessary overhead here: `logind`'s `PowerOff` method already *is* exactly this kind of narrow, purpose-built, auditable privileged interface. Building a bespoke broker to reimplement what `logind` already provides would be needless indirection.
- **New hardware (GPIO relay/smart power switch)** to achieve a genuine 0 W cutoff with fully automatic recovery — the only way to satisfy both halves of the original goal simultaneously, but out of scope: no such hardware exists in the current appliance design, and adding it is a hardware-design decision, not a software change.

## Hardware validation follow-up: the watcher must also count `"off"`, not just `"standby"`

Live testing on 2026-07-11 (real PartyBox 520, live battery, `idle_shutdown_minutes=2` for a fast repro) surfaced a gap in the initial decision above, before it ever reached the point of triggering a shutdown: the speaker didn't stay in `"standby"` — after enough idle time it auto-powered off past standby into a state where its BLE radio stopped advertising entirely. `DeviceManager` was left repeatedly scanning and failing to even discover it (`speaker_state: "off"`), while the Pi — confirmed by physical observation — remained fully powered throughout. This is the same battery-drain failure mode this ADR exists to prevent, just one step further along than the design above accounted for: the original watcher only counted idle time while `speaker_state == "standby"`, deliberately resetting on `"off"` on the reasoning that a dropped BLE link means the power source can no longer be reconfirmed. In practice, that meant the watcher would never fire at all once the speaker went fully dark — which, per this test, is a real and apparently reachable state, not just a theoretical one.

**Revised decision:** the watcher now counts idle time in *both* `"standby"` and `"off"`, using `last_known_on_battery` — the most recently confirmed `charging_status` reading, frozen at whatever it was when the BLE link dropped, rather than requiring a live reconfirmation. A confirmed reading is captured whenever one is available (during `"on"` or `"standby"`, whichever last had a successful battery/firmware probe); it persists across disconnects since a disconnected snapshot's `battery_status` is always `None`. Real activity (`speaker_state == "on"`) still unconditionally resets the idle clock.

**Accepted trade-off, made explicitly rather than by omission:** because `"off"` cannot be reconfirmed, a speaker that goes out of BLE range or gets moved onto a charger *during* an idle-on-battery spell will still be judged by the stale `last_known_on_battery` reading until (if ever) it reconnects — an appliance that's actually back on mains, or just temporarily out of range, could still be shut down. This was chosen over the alternative (require `"standby"` specifically) because that alternative was just shown, live, to never protect the battery at all in the one scenario that matters most — the original incident almost certainly went through this exact "off, BLE dark, Pi still fully powered" path, not a prolonged "standby" one. No staleness cutoff (e.g. "only trust `last_known_on_battery` for N minutes after disconnect") was added — the existing `idle_shutdown_minutes` threshold already bounds how long a stale reading can matter before it's acted on, and adding a second timeout was judged as unneeded complexity for what it would buy.

**Also caught live, a separate bug in the same test pass:** the watcher's idle/battery/last-known-power-source logic above worked correctly on the first real attempt — it fired the "no activity for 2 min" warning right on schedule — but `login1_dbus.power_off()` itself then crashed: `AttributeError: 'ProxyInterface' object has no attribute 'call_PowerOff'. Did you mean: 'call_power_off'?`. `dbus-fast` snake_cases a D-Bus method's PascalCase wire name (`PowerOff`) for its dynamically-generated `call_<method>` attribute; the original code passed `"PowerOff"` unchanged. This is not something either the unit tests or mypy could have caught: `login1_dbus.py`'s test double is a hand-rolled fake interface with whatever attribute name the test author gives it, and mypy has no visibility into `dbus-fast`'s runtime-introspection-generated attributes either — both simply validated the code's own (wrong) assumption back to itself. Only a real D-Bus connection exercises the actual attribute name `dbus-fast` generates. Fixed by passing `"power_off"`; both the fake and the real module now agree. The Supervisor's restart-on-crash policy meant this failed silently from the Portal's perspective (no crash visible outside `journalctl`) until the watcher's own state reset on restart and it stopped retrying — a live reminder that `Supervisor.health()` / `GET /api/v1/health/details` ([ADR-037](037-health-details-endpoint.md)) is where this kind of failure actually surfaces, not the primary health endpoint.

## Second follow-up: `"standby"` and `"off"` are not equivalent, and shouldn't share a threshold

Full end-to-end hardware validation succeeded after the fixes above — but review of the result raised a further question: does the full `idle_shutdown_minutes` threshold make sense for `"off"` at all? The two states differ in a way the original fix (above) elided by treating them identically:

- **`"standby"`** is recoverable *without touching the speaker* — the BLE control link stays up, and resuming playback wakes the amp back up on its own (observed PartyBox behaviour). A generous, user-configurable threshold exists specifically so an ordinary pause between songs, or a quiet moment at a party, doesn't take the whole appliance down.
- **`"off"`** means the BLE radio itself has gone dark. Nothing remote can recover that — the user has to physically touch the speaker's power button regardless of whether the Pi is still running. Once here, waiting out the same standby-oriented timeout buys no additional recoverability, only additional battery drain, for no benefit.

**Decision:** split the threshold. `"standby"` is still judged against the full Portal-configured `idle_shutdown_minutes`. `"off"` is judged against a new fixed, **non-configurable** constant, `_OFF_STATE_GRACE_SECONDS = 90.0` — not exposed as a product setting, since its purpose is purely to debounce a transient BLE reconnect blip (observed clearing on its own within under a minute during this same validation session — `DeviceManager`'s own reconnect loop already handles this without any intervention) rather than to represent "how long might the user still be listening," which is what the Portal setting means for `"standby"`.

The idle clock itself remains a single continuous counter that survives the `"standby"` <-> `"off"` transition — only the threshold compared against it changes with the current state. A speaker idle for a long time in standby that then drops to `"off"` fires on essentially the next check tick, since its already-accrued idle time dwarfs the short off-state threshold; there's no reason to make it wait out an *additional* grace period stacked on top of idle time it already accrued. Conversely, a speaker that drops straight from `"on"` to `"off"` (skipping a standby snapshot entirely — also observed live) starts its idle clock fresh at that point and must wait out the full 90 s.

Mains is unaffected by any of this: both branches still gate on `last_known_on_battery`, so a speaker confirmed on mains never arms either threshold, matching the product goal stated at the outset of this ADR (mains-connected → Companion runs indefinitely, no exceptions).

## Third follow-up: the Portal setting itself was removed

Once `"off"` had its own fixed, short threshold, a further question followed naturally: does the *value* of `idle_shutdown_minutes` — now only governing the `"standby"` branch — still do useful work as a Portal-exposed setting? In practice, real-world timing is dominated by however long the PartyBox's own undocumented on → standby → off progression takes, which Companion does not control and cannot observe in advance. The configured minutes value only changes the outcome if it happens to be *shorter* than that natural progression; otherwise the speaker reaches `"off"` on its own first and the fixed 90 s grace decides the timing regardless of what the Portal field said. A user-tunable number that mostly doesn't change the result is UI surface without a principled way for anyone to choose a value.

**Decision:** removed `idle_shutdown_minutes` from `PortalConfig` and the Settings sheet entirely. Both thresholds are now fixed internal constants — `_STANDBY_GRACE_SECONDS` (30 min) and `_OFF_STATE_GRACE_SECONDS` (90 s) in `companion/__main__.py` — following the same reasoning already applied to the off-state threshold in the second follow-up above. There is no remaining way to disable the feature; it is unconditionally on for every installation. `_idle_battery_shutdown()` no longer takes a `ConfigStore` dependency at all, since there is nothing left for it to read.

**Where 30 minutes specifically comes from:** the product goal in this ADR's Context section was stated as a "15–30 minutes" range, not a single number, because it was originally meant to be user-tunable. 30 min is the upper (more generous) end of that range, kept as the single fixed value once configurability was removed — a product choice carried forward from the original goal, not re-derived from anything measured. This is unlike `_OFF_STATE_GRACE_SECONDS`, which does have a concrete engineering lower bound (it must exceed the transient-reconnect duration observed on hardware). If real-world experience shows 30 min is wrong in either direction, it should be revised directly, not treated as load-bearing.

**Accepted trade-off:** an installation that genuinely wants a different standby allowance (e.g. a much longer or shorter one for an unusual usage pattern) has no supported way to change it short of editing the constant and redeploying. Judged acceptable: nobody had asked for that granularity, the existing knob's value barely mattered in the one scenario it was meant to protect (see above), and a config surface that doesn't reliably do what its label implies is worse than no config surface at all.
