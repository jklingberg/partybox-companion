# ADR-036: Push, Don't Poll — a Merged WebSocket Stream Across partyboxd and companion

**Status:** Accepted

---

## Context

The Portal redesign's roadmap (`docs/design/portal-redesign.md` §12, item 2) called for replacing the 2s pairing poll and slow reconciliation poll with push events, once the daemon could emit them — deferred at the time because only the BLE side (`speaker_state`, `connected`/`disconnected`) had push events at all ([ADR-033](033-speaker-standby-detection.md), [ADR-034](034-power-command-reconnect-wait.md)). Bluetooth Classic audio (`AudioService`), Spotify Connect (`SpotifyService`), and Bluetooth pairing (`PairingService`) were still poll-only: the Portal hit `GET /api/v1/audio` every 2 seconds during an active pairing attempt, and folded the rest into the general 15s reconciliation poll ([ADR-035](035-state-ownership-and-signal-pipeline.md)'s pipeline diagram, written just before this change, documents that pre-push state).

Two structural facts made this tractable now:

- `AudioService` already had its own `subscribe()`/`unsubscribe()` pub-sub (an `_AudioEventBus`, ADR-026) for internal use (gating `SpotifyService` on audio readiness). `DeviceManager` had an equivalent, independently-written `EventBus` for the same purpose. Adding push events to `SpotifyService` and `PairingService` would have meant writing that same broadcast-dispatcher class a third and fourth time.
- partyboxd's WebSocket endpoint (`GET /api/v1/events`) already existed and was hard-wired to `DeviceManager`'s bus alone. `SpotifyService`/`PairingService`/`AudioService` live one layer up, in `companion` — and per the project's layering (`partybox → partyboxd → companion`, ADR-004/005), partyboxd must not import companion's event types.

## Decision

### 1. Generalize `EventBus` once, at the third/fourth instance

Extracted the broadcast-dispatcher pattern (independently written twice already, in `DeviceManager` and `AudioService`) into a single generic `partyboxd.eventbus.EventBus[T]`. `DeviceManager`, `AudioService`, and the newly-instrumented `SpotifyService`/`PairingService` all now construct `EventBus[TheirEventType]()` rather than each hand-rolling the same subscribe/unsubscribe/emit logic. This follows the project's existing rule of thumb for generalizing narrowly-scoped helpers (ADR-032: introduce the shared abstraction "before the third or fourth [instance] lands, not before") — this is that third-and-fourth instance.

`EventBus` lives in `partyboxd` (not `companion`) specifically so companion can depend on it — the dependency only ever points one way.

### 2. `SpotifyService` and `PairingService` get `subscribe()`/`unsubscribe()`

Mirroring `AudioService`'s existing shape exactly:

- `SpotifyService.subscribe()` returns a queue pre-populated with the current `SpotifyStatusChanged` (running/active/device_name), then delivers one on every running/active transition. A new `_set_status()` helper centralizes what were previously several direct `self._running = ...` / `self._active = ...` assignments scattered across `_run_once`, `_infer_playback_state`, and `_terminate`, so no transition can be missed by a future edit.
- `PairingService.subscribe()` returns a queue pre-populated with the current `PairingProgressEvent` (state/error), then one per state transition. A `_set_state()` helper replaces the equivalent scattered assignments in `_do_pair()`.

Both follow [ADR-035](035-state-ownership-and-signal-pipeline.md)'s Rule 1 (single ownership) — the running/active or state/error pair is set in exactly one place now, not several.

### 3. partyboxd's WS endpoint accepts additional generic event sources

`make_ws_router`/`create_app` gained `extra_sources: Sequence[EventSource]`, where `EventSource` is a minimal structural `Protocol` (`subscribe() -> Queue[Any]`, `unsubscribe(queue)`) — satisfied by `DeviceManager` and by any `EventBus`-backed service without partyboxd importing a single companion type. `companion/__main__.py` passes `[audio, spotify, pairing]` when constructing the app.

Internally, each source gets one lightweight `_forward()` task that relays its queue into a single merged queue; the WS handler's main loop reads only that one merged queue (unchanged from before, apart from the source). This was chosen over recreating an `asyncio.wait()` over N per-iteration tasks each loop tick — one persistent forwarder per source is simpler to reason about and cheaper (no repeated task setup/teardown for sources that rarely emit).

### 4. New event types carry enough payload to update Portal state directly

`audio_changed` (`audio_ready`, `address`), `spotify_changed` (`running`, `active`, `device_name`), `pairing_progress` (`state`, `error`) — each one lets the Portal's `onEvent()` update `S.audio`/`S.spotify` directly from the event payload and re-derive the scene, with **no follow-up REST call**. This is the actual point of push over poll: not just faster notification that something changed, but skipping the round-trip entirely.

### 5. The Portal drops the 2s pairing poll; the general poll relaxes to 30s, safety-net only

`startPairPoll()` (2s `setInterval` against `GET /api/v1/audio`) is gone. `startPairing()` now sets a local `S.pairingActive` flag and a **180s** `setTimeout` safety net (matching the old poll's 90×2s give-up point); progress comes from `pairing_progress`/`audio_changed` WS events via `handlePairingProgress()`. This is the same shape as [ADR-034](034-power-command-reconnect-wait.md)'s `powerPending`: a client-local flag with no daemon source, bridging a real gap, cleared by the real signal when it arrives and by a bounded timer only as a last resort — see [ADR-035](035-state-ownership-and-signal-pipeline.md)'s "local-only state" rule.

The general reconciliation poll (`startReconcilePoll`) goes from 15s to 30s. It was already the safety net, never the primary update path for anything WS now covers; slowing it further only affects how quickly a *missed* event or a dropped-then-reconnected WS session catches up, not normal-path latency.

**Validated on hardware:** connected to the live `/api/v1/events` stream directly and observed `audio_changed`/`spotify_changed`/`pairing_progress` delivered on connect (seeded current state) and on real transitions (`audio_ready: false → true`, `spotify.running: false → true`) merged correctly alongside the pre-existing BLE-side events and heartbeat pings, over the same single connection.

## Operational properties

Reasoning below was requested during PR review as explicit documentation, not as a request for changes — each subsection states what's guaranteed, what isn't, and why the gap (where one exists) doesn't matter today.

**Ordering.** Guaranteed *per source only*. Within one source, FIFO is exact: a source's `EventBus.emit()` calls append to that connection's dedicated queue in call order, and `_forward`'s loop (`sink.put_nowait(await queue.get())`) drains and re-enqueues into the merged queue one at a time with no reordering possible — locked in by `test_events_from_one_source_preserve_emission_order`. *Across* sources there is no guarantee at all: `audio_changed`, `spotify_changed`, and `speaker_state_changed` come from independently-scheduled tasks (a BLE health-check loop, an A2DP subprocess check, a subprocess stdout reader), and their relative arrival order reflects whichever real-world signal actually completed first — not something this merge mechanism adds or removes. Where one event causally triggers another (e.g. `SpeakerStateChangedEvent` triggering `AudioService.recheck_now()` via `_recheck_audio_on_standby`, ADR-034), that specific pair is *effectively* ordered because the trigger only fires after the first event is fully processed — but that's a property of that specific wiring, not of the merge/fan-in itself.

The Portal's `onEvent()` handlers are robust to this by construction, not by defensive coding added for this concern: each handler for `audio_changed`/`spotify_changed`/`pairing_progress` overwrites only its own slice of `S` from that event's own payload, and `deriveScene()`/`handlePairingProgress()` are pure functions of *current* `S`, re-run in full on every relevant event — never of "what changed since the last event" or "what order things arrived in". Two orderings of the same event set converge to the same final `S` and the same rendered scene. This is a direct consequence of [ADR-035](035-state-ownership-and-signal-pipeline.md) Rule 1 (single ownership) plus the original redesign's "scene is derived, never stored" principle — reordering-robustness wasn't a separate design goal, it fell out of already having those two properties.

**A related, narrower race exists at connection start** (found while writing `test_ws_events_delivers_real_json_serializable_payloads`): `ws_events` creates the per-source forwarder tasks via `asyncio.create_task()`, but they don't actually reach `source.subscribe()` until the event loop schedules them — one tick after `ws_events` itself starts, which is itself one tick after the browser's connection is accepted. Any event a source emits inside that (sub-millisecond, but non-zero) window is invisible to *that* connection — not lost system-wide, just never delivered to a client that connected in the same instant a source happened to fire. Self-healing: `subscribe()` seeds current state immediately once it does run, and the next real transition or the 30s reconcile poll covers the rest. Not worth closing (would mean blocking `websocket.accept()` on every forwarder's subscribe() completing, adding a synchronization point for a gap that's already invisible in practice).

**Backpressure.** `EventBus.emit()` drops silently when a subscriber's queue is full (64 slots, per-connection). Every event type here (`audio_changed`, `spotify_changed`, `pairing_progress`) carries a full status snapshot, not a delta — so a dropped intermediate event just skips a transient step in the Portal's display (e.g. "Scanning…" straight to "Paired!"), not a corruption; the final state is whatever the *last* delivered event says, same as if nothing were dropped. The actual risk window (64 unconsumed events queued on one connection) would require a stalled WS client for far longer than any of these flows run — pairing alone gives up at 180s with at most a handful of real transitions in that time. Worth naming precisely: today's semantics drop the **newest** arrival when full, not the oldest — the reverse of "latest state wins," which would be the more obviously-correct policy for status-snapshot events. This is a real, identified inconsistency, deliberately not fixed here: with a 64-slot buffer against a realistic handful of events per flow, the two policies are indistinguishable in practice, and changing `EventBus`'s drop policy is a shared-infrastructure decision (it's used by `DeviceManager` too) that deserves its own review if a future event type ever emits often enough for the difference to matter — e.g. a live volume/VU-meter stream, not any event type that exists today.

**Seed events vs. duplicate transitions.** `AudioService._set_audio_ready()` and `SpotifyService._set_status()` no-op on an unchanged value (no emit fires). `PairingService._set_state()` didn't have the same guard — fixed in this PR (a small follow-up commit) for consistency, though it wasn't reachable as an actual bug: `_do_pair()`'s call sites never call it twice with the same state. Even hypothetically, a duplicate would have been harmless on the Portal: `handlePairingProgress()` gates its terminal-transition side effects (the toast, clearing the timer) behind `S.pairingActive`, which flips to `false` on the *first* delivery of a terminal state — a second, duplicate delivery of the same state finds `pairingActive` already `false` and returns immediately, so no double toast regardless.

**`EventSource` is deliberately not generic (`EventSource[T]`).** The one place that consumes it, `make_ws_router`'s fan-in, is *by design* type-erasing — it merges heterogeneous sources (`DeviceEvent`, `AudioServiceEvent`, `SpotifyEvent`, `PairingEvent`) into one `asyncio.Queue[Any]`, and the only thing done with an event is `dataclasses.asdict()` + JSON encoding, neither of which cares what `T` was. Parametrizing `EventSource[T]` would tie a source's declared type to `_forward`'s sink parameter, but since that sink is always `Queue[Any]` (intentionally, since it holds every source's events at once), the generic parameter would be erased back to `Any` at the only call site that exists — cosmetic, not load-bearing. Genericity would earn its keep the moment something needs to work with *one specific* source's event type in a type-checked way; nothing here does.

**Lifecycle/cancellation.** Traced explicitly: `_forward`'s only suspension point is `await queue.get()`, wrapped in `try/finally: source.unsubscribe(queue)` — cancellation can only take effect at that one await point (nowhere else in the loop body suspends), so `finally` always runs, and `EventBus.unsubscribe()` can't raise (`list.remove()`'s `ValueError` is caught). `ws_events`'s cleanup cancels every forwarder *then* awaits each one (in two separate loops) before returning, so by the time the handler exits, every unsubscribe has provably completed — no path leaves a queue subscribed, no hang (nothing in `_forward` can ignore cancellation indefinitely), and no leak under slow shutdown. The one benign race: if a source emits in the narrow window between `task.cancel()` being requested and the task actually processing it, that event can still ride through to `merged` before the `CancelledError` takes effect on the *next* loop iteration — harmless (the connection is tearing down anyway) and not a correctness issue for unsubscribe, which still runs regardless.

**Future extensibility.** The mechanism scales by flat addition today: a new source (AirPlay, provisioning, OTA, diagnostics) needs `subscribe()`/`unsubscribe()` (trivial via `EventBus[T]`) and one more entry in `__main__.py`'s `extra_event_sources=[...]` list — `make_ws_router` needs no change. The trigger for introducing a dedicated aggregator object instead of the flat list would be needing something the list can't express: per-source priority/QoS, conditional inclusion (feature flags), or transformation as events pass through (e.g. a server-side timestamp). None of that exists today; introducing the abstraction now would be paying for generality with no current consumer of it — the same "generalize at the third/fourth *real* instance" reasoning ADR-032 established and this ADR's own `EventBus` extraction already followed once.

**Testing strategy.** `packages/partyboxd/tests/unit/test_ws.py` avoids FastAPI's `TestClient` for WebSocket testing on purpose: `TestClient` runs the ASGI app in a separate thread with its own event loop, and pushing events into an `asyncio.Queue` owned by that other thread's loop from the test thread is not safe (`asyncio.Queue` has no `call_soon_threadsafe`-style cross-thread contract). The initial version of this test file covered `_forward` and the fan-in merge logic directly but stopped short of the actual `ws_events` handler, leaving no test of the real `dataclasses.asdict → send_json → JSON` path. Closed without reintroducing the threading problem: `APIWebSocketRoute.endpoint` is the original, unwrapped coroutine FastAPI registers (confirmed directly — decorating a route doesn't rebind the function), so it can be pulled off the router `make_ws_router` returns and called in-process against a minimal fake `WebSocket`, in the same event loop as the test. `test_ws_events_delivers_real_json_serializable_payloads` exercises this end-to-end, including a `StrEnum` field shaped like the real `PairingProgressEvent.state` (`PairingState`), to confirm it survives `asdict` + `json.dumps` as a plain string rather than an enum repr — the concrete case the review flagged.

## Consequences

**Benefits:**
- Pairing progress, Spotify status, and audio connectivity now reach the Portal in the same tick they happen, not up to 2s (pairing) or up to 30s (general poll) later.
- Removes a periodic subprocess-spawning `GET /api/v1/audio` call every 2 seconds for the duration of every pairing attempt (each `/api/v1/audio` fetch on the companion side may shell out to check A2DP status — see `AudioService._is_connected()`), reducing avoidable Bluetooth/D-Bus/subprocess chatter during exactly the window (active pairing) where the radio is already under the most pressure.
- No REST API surface changed — `GET /api/v1/audio`, `/spotify`, etc. still exist and still work for clients that don't use the WS stream (ADR-012 interoperability).

**Accepted trade-offs:**
- The Portal's near-real-time pairing feedback now depends on the WS connection specifically. If it drops mid-pairing, `ws.onclose`'s 5s reconnect plus the 30s reconciliation poll are the fallback — pairing was already a bounded, user-attended flow (with its own 180s give-up), so this is a acceptable degradation, not a silent one (the `S.pairingActive` timer still fires the "timed out" message even if no more events arrive at all).
- `EventSource`/`EventBus` add a small amount of structural ceremony (subscribe/unsubscribe symmetry, pre-populating the "current state" event) that any future companion service wanting push events must repeat — already an accepted cost for `AudioService` since ADR-026; this ADR just formalizes it as the pattern for new services too, rather than each reinventing it.

## Rejected alternatives

- **A second, companion-owned WebSocket endpoint** for audio/spotify/pairing events, leaving partyboxd's `/api/v1/events` untouched. Rejected: the Portal would need to manage two WS connection lifecycles (reconnect, auth, heartbeat) instead of one, for no benefit — nothing about these events requires a separate channel, and the design doc calls for one stream.
- **Recreate `asyncio.wait()` over all source queues every loop iteration**, instead of persistent per-source forwarder tasks feeding one merged queue. Rejected as more complex for no benefit: N tasks recreated every tick vs. N tasks created once per WS connection and left idling; the merged-queue approach is also easier to reason about (the main loop's logic is unchanged from before this ADR — it still just reads one queue).
- **Let companion own its own copy of `EventBus`** rather than sharing partyboxd's generic one. Rejected: would leave the exact duplication (now four services, not two) that motivated generalizing it in the first place, and companion already depends downward on partyboxd for other types (`SpeakerStateChangedEvent`, etc.) — adding `EventBus` to that existing dependency is free.
