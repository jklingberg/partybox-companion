# ADR-012: Interoperability Positioning and Legal Hygiene

**Status:** Accepted

---

## Context

partybox-companion develops an independent implementation of the Bluetooth protocol used by JBL PartyBox speakers. This work involves protocol analysis techniques — observing traffic, cross-referencing the official Android application — to understand how the device communicates.

Early documentation framed this work primarily as "reverse engineering." That framing is technically accurate but creates two problems:

**Problem 1: It misrepresents the project's purpose.** The project exists to give users open integration with hardware they own. Protocol analysis is one engineering technique used along the way, not the goal. Centering the project identity on reverse engineering suggests the purpose is to study or expose JBL's software, rather than to build a useful independent implementation.

**Problem 2: It invites the wrong contributors and the wrong questions.** A project that presents itself as a reverse engineering effort will attract people who want to extract proprietary software, not people who want to build an open appliance. It also raises unnecessary questions about intent that obscure the legitimate purpose.

Open interoperability has strong legal footing. EU Directive 2009/24/EC and equivalent statutes in many jurisdictions explicitly permit analysis of software for the purpose of interoperability. Projects like Samba, Wine, and numerous Bluetooth implementations have been developed on this basis. The legal position of an interoperability project is strongest when the implementation is clearly independent — not a reproduction of the original software.

## Decision

The project presents itself as an **independent interoperability implementation** rather than a reverse engineering project.

Concretely:

- Public-facing documentation (README, Vision) leads with what the project enables, not how the protocol was discovered
- Reverse engineering documentation is retained in full but positioned as developer documentation — part of the contributor workflow, not the project identity
- The repository maintains a clear separation between user documentation, developer documentation, and protocol analysis notes
- A disclaimer is added to the README clarifying the project is not affiliated with JBL or HARMAN
- Contributors are guided toward original implementation based on observed behaviour

### Legal hygiene for contributors

The repository and its documentation encourage the following practices:

**Never commit to the repository:**
- APK files from the JBL app or any proprietary application
- Source code generated by decompilation tools (JADX, jadx-gui, apktool, etc.)
- Copyrighted application resources (images, strings, binary assets)
- Any file extracted from a proprietary application

**When documenting protocol behaviour:**
- Document observations from wire traffic — byte values, message structure, timing
- Do not transcribe or paraphrase proprietary source code
- Record what the protocol does, not how the proprietary implementation does it

**When implementing:**
- Write original code derived from protocol observations, not from proprietary source
- The standard is: if a lawyer asked "did you copy their code?", the honest answer is no

Raw research artifacts (APK files, JADX exports, captures) live in `research/` and are excluded from version control by `.gitignore`. The `.gitignore` entries exist to prevent accidental commits of proprietary material, not merely to reduce repository size.

## Consequences

**Benefits:**
- The project's purpose is immediately legible to anyone who finds the repository — including JBL engineers, potential contributors, and users.
- Legal positioning is stronger. An independent implementation that documents its observations and writes original code is clearly protected interoperability work.
- The project attracts contributors who want to build the appliance, not contributors who want to redistribute the JBL app.

**Accepted trade-offs:**
- Protocol analysis work is now positioned as contributor-facing rather than user-facing. Some contributors who found the project through its RE work may need to navigate to the developer docs to find that material.
- The framing requires some discipline across contributors — the legal hygiene guidelines are advisory, not enforced by tooling.

**What does not change:**
- The content of the reverse engineering documentation. All protocol findings, the guide, the discoveries log, and the open questions remain in full.
- The use of JADX and traffic capture as development tools. These are standard interoperability analysis techniques and remain part of the contributor workflow.
- The project's technical approach. This ADR changes how the work is described, not what the work is.

**Relationship to other decisions:**
- See [ADR-001](001-project-vision.md) for the appliance-over-library positioning that gives users, not developers, primacy.
- See [ADR-010](010-sdk-scope.md) for the scope decision that keeps the SDK to hardware-unique features, reinforcing that this is an integration project rather than an attempt to replicate the full JBL app.
