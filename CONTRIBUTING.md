# Contributing

Thank you for your interest in partybox-companion.

This guide covers how to set up a development environment, the rules that keep the codebase maintainable, and how to contribute different kinds of changes.

For the architectural reasoning behind the project structure, see [docs/architecture.md](docs/architecture.md) and [docs/adr/](docs/adr/).

---

## Setup

```bash
# Install uv (https://docs.astral.sh/uv/)
curl -LsSf https://astral.sh/uv/install.sh | sh

# Clone the repository
git clone https://github.com/jklingberg/partybox-companion.git
cd partybox-companion

# Install all packages and dev dependencies
uv sync --all-extras

# Install pre-commit hooks
uv run pre-commit install
```

---

## Running checks

```bash
# Format
uv run ruff format .

# Lint
uv run ruff check .

# Type check (run from each package directory)
cd packages/partybox  && uv run mypy src/; cd ../..
cd packages/partyboxd && uv run mypy src/; cd ../..
cd packages/companion && uv run mypy src/; cd ../..

# Tests (no hardware required)
uv run pytest packages/partybox/   -m "not hardware"
uv run pytest packages/partyboxd/  -m "not hardware"
uv run pytest packages/companion/  -m "not hardware"
```

## Package boundaries

The project has three packages. The dependency direction is one-way:

```
partybox  ←  partyboxd  ←  companion
```

**`partybox`** must never import from `partyboxd` or `companion`. Its only runtime dependency is `bleak` (the BLE GATT transport).

**`partyboxd`** may import from `partybox`. It must not implement protocol logic.

**`companion`** may import from `partyboxd` and `partybox`. It must not reach into Bluetooth or protocol internals.

Within `partybox`, the internal dependency order is also strict:

```
bluetooth/ → protocol/ → device/
```

`protocol/` imports only from `bluetooth/`. `device/` imports only from `protocol/`. Nothing imports from a layer above it.

## Legal hygiene

partybox-companion is an **independent interoperability implementation**. This distinction matters for all contributors.

**Never commit to this repository:**
- APK files from the JBL app or any other proprietary application
- Source code generated by decompilation tools (JADX, apktool, etc.)
- Copyrighted application resources — images, strings, binary assets — from proprietary software
- Any file extracted or derived from a proprietary application

**When documenting protocol behaviour:**
- Document your *observations* — byte values on the wire, message structure, timing, response sequences
- Do not transcribe or paraphrase proprietary source code, even partially
- Record what the protocol does, not how the proprietary implementation achieves it

**When implementing:**
- Write original code derived from your protocol observations
- The goal is an independent implementation, not a translation of the JBL app
- If a reviewer asked "did you copy their code?", the honest answer should be no

Raw research artifacts (APK files, JADX exports, captures, scripts) live in `research/` and are excluded from version control. The `.gitignore` entries for those directories exist to prevent accidental commits of proprietary material. See `research/README.md`.

## Adding a protocol command

Most new protocol commands follow the same workflow.

See [docs/developer-guide.md](docs/developer-guide.md#adding-a-protocol-command) for the complete process.

In summary:

1. Locate the feature in the decompiled Android application (JADX) to identify the relevant classes, opcode and payload.
2. Confirm the behaviour by capturing Bluetooth traffic with **nRF Connect** (HCI snoop log) while exercising the feature in the official app.
3. Document the command in [docs/reverse-engineering/protocol.md](docs/reverse-engineering/protocol.md), including:

   * opcode
   * payload format
   * expected response
   * confidence level
   * any open questions
4. Implement the command in the `partybox` SDK by adding the appropriate message type and updating the parser/serializer.
5. Expose the functionality through the appropriate device capability rather than model-specific logic.
6. Verify the implementation against a real PartyBox and capture representative packets for regression tests.
7. Add unit tests using captured packet fixtures so protocol parsing can be tested without physical hardware.
8. Once the SDK supports the command, expose it through higher-level components such as `partyboxd`, the REST API, or the Companion Portal as appropriate.

Design principle: When possible, derive protocol semantics from the decompiled Android application and use Bluetooth captures only to validate behaviour and timing. This provides more context than captures alone. Implement behaviour independently based on your observations rather than translating proprietary code.

## Adding a new capability

1. Create `packages/partybox/src/partybox/device/capabilities/<name>.py` — a plain class following `power.py` as the template (there is no shared base class)
2. Add the `@property` to `device/partybox.py` — typed `<Name>Capability | None` if the capability is optional
3. Export from `partybox/__init__.py` if it is part of the public API
4. Add tests

## Hardware tests

Tests that require a real PartyBox are marked `@pytest.mark.hardware`. They are never run in CI.

```bash
PARTYBOX_ADDRESS=AA:BB:CC:DD:EE:FF uv run pytest packages/partybox/ -m hardware -v
```

## Commit messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
feat(protocol): add EQ band command
fix(bluetooth): handle reconnect race condition
feat(api): add WebSocket event stream
docs(reverse-engineering): document volume response format
test(device): add MockBackend reconnect scenario
```

Scopes: `bluetooth`, `protocol`, `device`, `capabilities`, `api`, `services`, `config`, `webui`, `docs`, `ci`

---

## Pull requests

- One feature or fix per PR
- Tests required for new protocol features (use real BT captures as fixtures)
- `mypy --strict` must pass for all packages
- All CI checks must be green
- Link the related issue if one exists

---

## Contributing protocol documentation

If you have a PartyBox model that is not yet documented, your captures are valuable even if you cannot write Python. Open an issue with the capture file attached and describe what actions you triggered. See [docs/reverse-engineering/guide.md](docs/reverse-engineering/guide.md).
